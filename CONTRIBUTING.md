# Contributing to GitCat

For install/build/test commands, see the [Development](README.md#development) section of the README. This file is about getting *oriented* in the codebase before you start reading or changing it.

## Look at the history before the code

Before diving into any unfamiliar codebase, a few `git log` queries tell you more than the first hour of reading source ever will — which files are the real center of gravity, and whether that's by design or by accident. (Inspired by [this post](https://piechowski.io/post/git-commands-before-reading-code/); worth reading in full.)

Most-changed files in the last year:

```bash
git log --format=format: --name-only --since="1 year ago" | sort | uniq -c | sort -nr | head -15
```

Run today, that's:

```
61  index.html
45  src/legacy/main.ts
42  src-tauri/src/lib.rs
38  src/ipc/bindings.ts
34  src/main.ts
25  src/islands/sidebar/Sidebar.svelte
20  src/islands/sidebar/sidebar.svelte.ts
18  src-tauri/src/menu.rs
```

**That's expected, not a red flag** — each one is a real architectural hub, not an accumulation of patches-on-patches:

- **`index.html`** / **`src/legacy/main.ts`** — the original single-file vanilla app: the canvas commit-graph renderer, drag-and-drop, and (in `index.html`) every shared CSS class the Svelte islands themselves reuse. Almost any new UI feature touches one or both, by design (see `src/legacy/bridge.ts`'s own doc comment for how islands reach into this layer).
- **`src-tauri/src/lib.rs`** — the single `collect_commands!` list every Tauri command is registered in. Every new backend command, however small, adds one line here.
- **`src/ipc/bindings.ts`** — auto-generated (see below); churns on every backend change by design, never by hand.
- **`src-tauri/src/menu.rs`** — the native app menu; every Tools-menu/⌘K-reachable feature adds a menu item and an id here.

If you're about to touch one of these files and it looks unfamiliar or sprawling, that's normal for this codebase's shape — read the surrounding doc comments (this repo leans heavily on inline "why" comments over separate design docs) before assuming it needs a refactor.

This project is also very young — every commit so far is from July 2026 — so "history" here means days, not years. The technique still holds; there's just less of it yet.

## Where things live

- `src-tauri/src/` — Rust core. `git2` for reads, the `git` CLI for writes (every mutation snapshots first via the Safety Manager). One file per feature area (`git_bisect.rs`, `submodule.rs`, `tool_settings.rs`, …), each with its own module-doc comment explaining its own design decisions.
- `src/islands/<name>/` — one Svelte 5 "island" per feature (a `.svelte.ts` runes controller + its `.svelte` view, some with no view at all — see `applypatch.svelte.ts`/`forcepush.svelte.ts` for the "no modal, just fires a command" shape).
- `src/legacy/` — the original vanilla-canvas app (`main.ts`, `@ts-nocheck`) and `bridge.ts`, the live-binding seam islands use to reach into it.
- `src/ipc/bindings.ts` — auto-generated by tauri-specta; never hand-edit, regenerate with `cd src-tauri && cargo test export_bindings`.
- `docs/` — the [VitePress](https://vitepress.dev) marketing/docs site, deployed separately from this file.
